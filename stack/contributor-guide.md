# Stack Contributor Guide

## 前提

- フロントは `React` で統合する（単一 Web アプリ）
- 共同開発者は、担当スタックの **知見・サンプル・契約案・検証コード** を `stack/` 配下に整理する
- `stack/` の実装は学習・検証・記事素材が目的であり、本番バックエンドへの統合は自動ではない
- 本番採用には ADR とメイン開発者のレビューが必要

## 整理するもの

- `README.md`: そのスタックで何を学ぶか・何を検証するか・どんな記事素材を作るか
- `api/contract.md`: 将来統合時の参考 API 契約案（学習用の契約整理 — 本番契約ではない）
- `project/`: 実行可能な最小構成（学習用・PoC 用）
- `notes/workflow.md`: 環境構築メモ、学習ログ、ハマりどころ
- `examples/` または `templates/`: サンプル入出力、テンプレート（任意）

## 参考にするディレクトリ

共同開発者が新しい言語ディレクトリを整えるときは、`stack/language/go/` を基準例として参照する。

特に以下を揃えることを意識する。

- 一覧ページ向けの短い要約
- 初学者向けの記事素材
- 中級者・開発者向けの記事素材
- 実行可能な最小 `project/`
- 比較や記事本文で引用しやすい `examples/`
- 将来統合時の参考になる `api/contract.md`

Go ディレクトリは、共同開発者向けに「どの粒度で資料とサンプルを整理するか」を示す参考実装として `main` に配置している。

## api/contract.md を書く際の観点

フロントとの将来的な統合を念頭に、以下を整理しておくと有用である。

- エンドポイント候補と HTTP メソッド
- リクエスト / レスポンス形式（JSON）
- 認証方式の案（実装依存部分は明記する）
- エラー時のレスポンス形式

これはあくまで **将来統合時の参考 / 学習用の契約案** であり、フロントがこの API を直接叩くことを前提にしていない。

## 記事素材として使えるもの

- 言語・フレームワーク・DB の特徴の比較コード
- 実際に動く最小サンプル（ハンズオン素材）
- ハマりどころ・解決策のメモ
- パフォーマンス計測結果、ベンチマーク比較

## 新しいスタックを追加するには

```bash
./scripts/new-stack.sh <category> <stack-name>
```

スクリプトがテンプレートを自動生成する。
