# 実装前タスク分割・担当割り振り・開発フロー

## 目的

このドキュメントは、`tech-atrium-wiki` の実装着手前に、担当領域の切り分け、並行開発しやすい単位、レビュー経路、コミット方針を整理するための運用草案である。

メイン開発者が React フロントと全体統合を主導しつつ、共同開発者が `stack/` 配下で技術スタック別の学習・検証・PoC コードを継続的に追加する前提で管理する。

---

## MVP バックエンド原則

**MVP では、本番バックエンドを単一の Web アプリ / 単一の主要バックエンドに統一する。**

デプロイ方式（案 A / B / C）にかかわらず、この原則は変わらない。`stack/` 配下の実装は、本番バックエンドとしてただちに統合するものではない（詳細: `docs/adr/0001-backend-policy.md`）。

---

## 前提

- フロントエンドの基盤は `React` を中心に `Next.js` とする
- 共同開発者のコードは `stack/` 配下で管理し、学習・検証・記事素材・将来候補として扱う
- ドキュメント、仕様、サンプルコード、契約案を分離して管理する
- 各担当者は自分の専門領域を持つが、レビューは領域横断で行う
- コミットは可能な限り小さく分け、履歴から変更意図が追える状態を維持する

---

## 役割

| 役割 | 主な責務 |
|---|---|
| メイン開発者 | 全体設計、最終判断、統合、マージ方針の管理 |
| React 担当 | UI 基盤、画面実装、コンポーネント設計 |
| Python 担当 | **学習・検証**: Python サンプル、データ整形スクリプト、記事素材の検証補助 |
| Dart / Flutter 担当 | **学習・検証**: モバイル観点の UI 知見、Flutter/Dart 解説記事・比較サンプルの整備 |
| Rust 担当 | **学習・検証**: 高性能処理の検証 PoC、CLI 補助ツール試作、記事素材の整備 |
| コンテンツ担当 | MDX 記事、比較データ、タグ、用語整備 |
| レビュー担当 | PR 確認、仕様整合、差分の品質担保 |

> **注意**: Python / Dart / Flutter / Rust 担当は **本番 API の担当ではない**。`stack/` 配下での学習・検証・記事作成が主な役割であり、本番バックエンドとしての統合は ADR 審査を経てから判断する。

## 想定アサイン

現時点では以下のような割り振りを想定する。

| 領域 | 想定人数 | 主担当 |
|---|---|---|
| 全体統合 / リリース管理 | 1 | メイン開発者 |
| React / Next.js | 1-2 | フロントエンド担当 |
| Python（学習・検証・補助） | 2-3 | データ整形・記事素材担当 |
| Dart / Flutter（学習・検証） | 1 | モバイル観点担当 |
| Rust（学習・検証・補助ツール） | 1 | 高性能処理・ツール担当 |
| MDX / コンテンツ整備 | 1-2 | コンテンツ担当 |
| レビュー横断 | 1-2 | メイン開発者 + サブレビュー担当 |

備考:
`Python 担当が複数人、Dart / Flutter 担当が 1 人、Rust 担当が 1 人` という体制を起点に、参加者増加に合わせて細分化する。

## タスク分割方針

### 1. 共通基盤

- リポジトリ構成の確定
- `Next.js` 初期構成
- デプロイ方式の選定（案 A / B / C の決定）
- 共通 UI ルール
- Lint / Format / Test の最低限整備
- `stack/` の命名規約とサンプルコード規約整備

### 2. コンテンツ基盤

- MDX 記事構造の定義
- カテゴリ設計
- 比較対象データのスキーマ定義
- タグ、難易度、メタデータの標準化

### 3. 機能別実装

- 認証（採用方式: Supabase Auth または Auth.js）
- 記事表示
- 比較エンジン
- コメント
- 修正提案
- トレンド検索

### 4. 学習・検証・補助

- Python によるデータ変換スクリプト（`stack/language/python/`）
- Rust による検証用 CLI や高速処理の試作（`stack/language/rust/`）
- Flutter 観点でのモバイル表示チェック項目整理（`stack/language/dart/`）
- 各スタックの技術記事・比較サンプルの作成

## 担当ごとの作業範囲

### メイン開発者

- ディレクトリ構成と責務分離の確定

- 複数担当の変更が衝突した際の調整

- 共通レイアウト
- 比較 UI
- 記事 UI

| 区分 | 主な担当 |

- MDX 記事追加
- 用語統一

- トレンドタグ整備

## stack/ 成果物の本番採用フロー

`stack/` または `spike/` の成果物を本番バックエンドに採用する場合は、以下のプロセスが必要である。

```
spike/stack/ での検証
        ↓
技術的妥当性の確認（認証・認可・データ整合性・運用担当）
        ↓
ADR に採用理由を記録（docs/adr/）
        ↓
メイン開発者のレビューとマージ判断
        ↓
本番統合
```

ADR なしに `stack/` の実装を本番に直接持ち込まないこと。

## 開発フロー

### 1. 着手前

- `docs/specs/requirements.md` と `docs/specs/design.md` を確認する
- 必要なら `stack/README.md` と `stack/contributor-guide.md` を確認する
- 着手タスクを issue または `docs/operations/active-work.md` へ記載する
- 担当者とレビュー担当を明確にする
- 変更対象ディレクトリを宣言する

### 2. 実装中

- 1 タスク 1 ブランチを原則とする
- 大きな実装は `draft PR` または進捗共有用コミットで早めに見せる
- 他担当の領域に触る場合は、先に責務衝突を確認する
- データ定義変更と UI 変更は、可能ならコミットを分離する

### 3. レビュー前

- 仕様との差分を自分で確認する
- 影響範囲を PR 説明に書く
- 必要ならスクリーンショットやサンプルデータを添える
- コミットメッセージが変更意図を説明していることを確認する

### 4. レビュー

- まず仕様整合
- 次に責務分離
- 次に命名、可読性、保守性
- 最後に将来の拡張余地を確認する

### 5. マージ

- レビュー指摘反映後にマージする
- マージ前に競合解消を済ませる
- マージ後は必要に応じて設計ドキュメントも追随更新する

## ブランチ運用

| 種別 | 命名例 | 用途 |
|---|---|---|
| 機能追加 | `feature/auth-login` | 新機能 |
| ドキュメント | `docs/task-split` | 要件、設計、運用文書 |
| 修正 | `fix/comment-validation` | 不具合修正 |
| リファクタ | `refactor/comparison-schema` | 責務整理 |
| 検証 | `spike/rust-cli-prototype` | 試作、技術検証（本番統合前提ではない） |
| ADR | `adr/0002-auth-method` | アーキテクチャ決定記録 |

## コミット方針

- 1コミット1意図を原則とする
- UI 修正、データ修正、ドキュメント修正をなるべく分離する
- 自動整形だけの差分は機能差分と分ける
- 日本語だけ、英語だけではなく、できるだけ英語 / 日本語の両方で意味が追えるようにする

コミットメッセージ例:

```text
feat: add article level tabs / 記事レベル切替タブを追加
docs: define task ownership / タスク担当範囲を定義
fix: handle empty comments / 空コメントを拒否
refactor: split comparison schema / 比較スキーマを分割
spike: prototype rust cli for data validation / Rust CLIでのデータ検証を試作
adr: record backend policy decision / バックエンド方針をADRに記録
```

## レビュー観点

- 要件定義と矛盾していないか
- 設計書の責務分離に沿っているか
- MVP バックエンド原則（単一構成）に反していないか
- 他担当者が追従しやすい構成か
- ハードコードされた担当依存ロジックがないか
- 後から参加した開発者が読める粒度になっているか

---

## 先行して確定したい事項

- デプロイ方式（案 A / B / C）の選定
- Python 担当の正式人数
- React 担当を 1 人にするか 2 人にするか
- Dart / Flutter 担当の責務をモバイル UI レビュー中心にするか、将来アプリ試作まで含めるか
- Rust 担当の責務を補助ツールまでに留めるか、比較ロジック本体の候補実装まで含めるか
- issue 管理を GitHub Issues で行うか、`docs` 内のタスク表で暫定管理するか

---

## 更新ルール

- 担当者が増えたら、人数だけでなく責務の分割単位も更新する
- 実装フェーズに入ったら、想定アサインを実アサインに置き換える
- 運用が固まったら、暫定ルールを削って正式フローだけ残す
- デプロイ方式が確定したら、方式固有の設定手順を追記する
