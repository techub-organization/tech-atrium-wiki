# ADR-0001: MVP バックエンド構成方針

## ステータス

採用

## コンテキスト

tech-atrium-wiki は開発部全員参加型のプロジェクトであり、複数の共同開発者が `stack/` 配下でさまざまな言語・フレームワーク・DB の実装案を持ち寄る設計になっている。

一方で、複数のバックエンドを最初から本番統合すると以下の問題が生じる。

- 認証・認可の責務が分散し、整合性の維持が困難になる
- 各サービス間のデータ整合性・障害時の挙動・監視が複雑になる
- 運用担当者が不明確になりやすく、障害対応が滞る
- MVP のリリース速度が低下する

また、開発部員のフルスタック育成という観点から、`stack/` の実装はまず **学習・検証・記事素材** として活用し、本番統合は段階的に判断することが適切である。

## 決定

**MVP では、本番バックエンドを単一の Web アプリ（Next.js）/ 単一の主要バックエンドに統一する。**

`stack/` 配下の各言語・フレームワーク・DB 実装は以下を目的とした作業場として位置づける。

- 学習サンプル（各言語・フレームワーク・DB の実装例）
- 技術記事の材料（実装パターンの解説素材）
- API 契約や実装比較の検証（Spike / PoC）
- 将来の本番切り出し候補の蓄積

## 理由

- 単一アプリ構成にすることで、認証・認可・データ整合性の管理を一元化できる
- MVP のリリース速度を最大化できる
- `stack/` の学習価値は本番統合なしでも十分に活かせる
- 将来的に分割が必要になった場合でも、ADR で理由を記録したうえで段階的に移行できる

## 本番バックエンドを分割する条件

`stack/` の実装を本番バックエンドに追加する場合は、以下をすべて満たす必要がある。

1. 単一 Web アプリ構成では解決しづらい明確な理由がある
2. 認証・認可・データ整合性の責務が明確である
3. 運用担当者が決まっている
4. 新しい ADR に採用理由を記録する
5. API 契約、障害時の挙動、監視方法が定義されている

## 却下した選択肢

**各共同開発者の `stack/` 実装を MVP から本番統合する案**

- Pro: 担当者のモチベーションが上がりやすい、フルスタック体験が早い段階で得られる
- Con: 認証・認可・データ整合性の管理が複雑になる、障害時の対応者が不明確、MVPのリリース速度が低下する
- 判断: 育成目的は `stack/` での検証・学習で十分に満たせるため、不採用

## 影響

- `stack/` の実装は本番 API として直接利用されないことを全共同開発者が理解する必要がある
- `docs/guides/collaborator-backend-guide.md` にこの方針を明記する
- 将来の本番統合判断は別途 ADR を起票して行う
